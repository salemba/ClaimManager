#Role:
Tu es un développeur senior TypeScript spécialisé en systèmes de gestion de sinistres assurance.
Tu produis du code production-grade, strictement typé, sans mutation d'objets,
avec des erreurs métier typées. Tu commentes uniquement les décisions non triviales.

#Context:
Entité Claim : id, adjusterId, status (enum), amount (€), blockedSince (timestamp), createdAt
Workflow ordonné : RECEIVED → UNDER_REVIEW → PENDING_DOCS → EXPERT_ASSIGNED → VALIDATED → CLOSED
Seul un utilisateur avec le rôle SUPERVISOR peut déclencher cette action.

#Business Rules:
L'action est autorisée si AU MOINS UNE des conditions suivantes est vraie :
- (now - claim.blockedSince) > 48h
- claim.amount > 10 000 €

#Task:
Implémente la fonctionnalité de réassignation manuelle superviseur avec les 4 livrables suivants :

1. Fonction canForceReassign(claim: Claim, now: Date): { allowed: boolean; reasons: string[] }
   → Évalue les conditions et retourne les raisons déclenchantes pour affichage dans la modale.

2. Service forceReassignClaim(claimId, newAdjusterId, supervisorId) :
   - Vérifier les conditions via canForceReassign (erreur métier typée si non remplies)
   - Récupérer l'état suivant dans le workflow (erreur si CLOSED : aucun état supérieur)
   - Mettre à jour le sinistre (adjusterId + status) sans muter l'objet original
   - Persister un ForceReassignAudit (supervisorId, timestamp, motif, état avant/après, adjusterId avant/après)
   - Notifier nouveau + ancien gestionnaire dans un try/catch non bloquant
   - Retourner le sinistre mis à jour

3. Endpoint REST :
   PUT /claims/:id/force-reassign
   Body : { newAdjusterId: string } — Auth : Bearer token rôle SUPERVISOR
   200 ok | 403 rôle insuffisant | 422 conditions non remplies | 404 introuvable

4. Tests unitaires :
   - Bloqué > 48h → autorisé
   - Montant > 10 000 → autorisé
   - Les deux conditions vraies → autorisé, les deux raisons remontées
   - Aucune condition → rejeté avec message explicite
   - État CLOSED → erreur "aucun état supérieur possible"

#Constraints:
- TypeScript strict
- Pas de mutation directe de l'objet claim
- Erreurs métier = classes typées (jamais des strings brutes)
- Audit persisté même si les notifications échouent (try/catch séparé)
- Repositories injectables via interface (testabilité)
- Commenter uniquement les décisions non triviales
